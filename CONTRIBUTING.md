# Вклад в mock_api

Спасибо за интерес к развитию `mock_api`.

## Быстрый старт

1. Сделайте fork и клонируйте репозиторий.
2. Установите зависимости:
   ```bash
   uv sync --locked --dev
   ```
3. Создайте локальный env-файл:
   ```bash
   cp .env-example .env
   ```
4. Примените миграции:
   ```bash
   uv run alembic upgrade head
   ```
5. Запустите приложение:
   ```bash
   uv run fastapi dev app/main.py
   ```

## Процесс разработки

1. Создайте ветку от `main`.
2. Вносите сфокусированные изменения.
3. Добавляйте или обновляйте тесты для всех изменений поведения.
4. Запускайте локальные проверки:
   ```bash
   uv run ruff check .
   uv run mypy .
   uv run pytest
   ```
5. Откройте pull request с понятным описанием проблемы и кратким summary изменений.

## Рекомендации для pull request

- По возможности держите PR небольшими и одноцелевыми.
- Если меняются схемы/данные, добавляйте migration notes.
- Любое изменение поведения API отражайте в `README.md` и `CHANGELOG.md`.
- Если меняется API-контракт, перегенерируйте OpenAPI:
  ```bash
  uv run python scripts/export_openapi.py --output docs/openapi/openapi.v1.json
  ```

## Стиль commit-сообщений

Рекомендуется легковесная конвенция:

- `feat: ...` для новой функциональности
- `fix: ...` для исправления багов
- `docs: ...` для документации
- `test: ...` для тестовых изменений
- `chore: ...` для инфраструктуры/инструментов

## Сообщения о багах

Пожалуйста, укажите:

- ожидаемое поведение
- фактическое поведение
- шаги воспроизведения
- детали окружения (OS, версия Python, версия Docker)
- логи или traceback (если есть)

## Нужна помощь?

Создайте GitHub issue с меткой `question` и кратким описанием вашего сценария.
